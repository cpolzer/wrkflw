# Implementation Plan: CI Auto-Versioning via Conventional Commits

**Branch**: `006-ci-auto-versioning` | **Date**: 2026-06-19 | **Spec**: [spec.md](spec.md)

## Summary

Add a fully-automated semantic versioning pipeline driven by conventional commits. `release-please` (Google) runs on every push to `main`, accumulates version-triggering commits into a release PR, and publishes a GitHub Release when that PR is merged. A separate publish workflow then builds all three Docker images (api-service, worker-service, ui), scans each with Trivy for CVEs, and verifies OCI version labels with skopeo. No images are pushed to a registry in this phase — the pipeline validates the full release shape at zero cost and is structured for a one-line activation of a real push when a registry is provisioned.

**Performance Goals**: Release PR opened within 5 min of merge; all three images built and scanned within 15 min of release PR merge (SC-001, SC-003)

**Constraints**: No registry credentials required; no external paid services; `docker-compose.yml` is out of scope (developer-only tooling)

## Phase 0: Research

See [research.md](research.md) — all decisions resolved, no NEEDS CLARIFICATION items.

**Key decisions summary**:

| Decision | Choice | Key reason |
|----------|--------|------------|
| Versioning tool | release-please v4 | Only tool with built-in release PR review gate |
| Version source | `gradle.properties` + `ui/package.json` | Gradle reads it automatically; npm convention |
| Release type | `simple` at root | Single shared version across monorepo |
| Docker scanner | Trivy (`aquasecurity/trivy-action`) | Native GitHub Action, SARIF output, free |
| Image verifier | Skopeo (apt on ubuntu-latest) | OCI-native label inspection without container start |
| Publish trigger | `release: types: [published]` | Clean separation of versioning and artifact steps |
| Tag format | `vMAJOR.MINOR.PATCH` (git) / `MAJOR.MINOR.PATCH` (image) | release-please default; OCI convention |

## Phase 1: Design

### Workflow 1 — release-please.yml

Trigger: `push` to `main`
Permissions: `contents: write`, `pull-requests: write`

```
push to main
  └─► googleapis/release-please-action@v4
        ├─ if version-triggering commits exist → open/update Release PR
        │    PR title: "chore(main): release 1.2.0"
        │    PR body: CHANGELOG section for this version
        │    PR changes: gradle.properties, ui/package.json, .release-please-manifest.json, CHANGELOG.md
        └─ if no triggering commits → no-op
```

### Workflow 2 — publish.yml

Trigger: `release: types: [published]` (fires when release-please merges the Release PR)

```
release published (tag: v1.2.0)
  └─► job: build-and-verify (matrix: api-service, worker-service, ui)
        │
        ├─ checkout
        ├─ set VERSION = tag[1:] (strip 'v' prefix → "1.2.0")
        │
        ├─ docker build
        │    --build-arg VERSION=$VERSION
        │    --tag wrkflw/$SERVICE:$VERSION
        │    --tag wrkflw/$SERVICE:latest
        │    --file <service>/Dockerfile
        │    .
        │
        ├─ trivy image
        │    --exit-code 1
        │    --severity CRITICAL,HIGH
        │    --format sarif
        │    wrkflw/$SERVICE:$VERSION
        │
        └─ skopeo inspect docker-daemon:wrkflw/$SERVICE:$VERSION
             | jq -e '.Labels["org.opencontainers.image.version"] == "$VERSION"'
             # non-zero exit → pipeline fails
```

### Version Injection per Artifact

| Artifact | Mechanism | Where version surfaces |
|----------|-----------|----------------------|
| `api-service` | `ARG VERSION` in Dockerfile → `LABEL org.opencontainers.image.version` | OCI label; app can read via `System.getenv("APP_VERSION")` if ENV is also set |
| `worker-service` | Same as api-service | OCI label |
| `ui` | `ARG VERSION` + `ENV VITE_APP_VERSION` in Dockerfile build stage | Available as `import.meta.env.VITE_APP_VERSION` in Vue code; OCI label on final image |

### Frontend Dockerfile Shape

```dockerfile
# Stage 1: build
FROM node:22-alpine AS builder
WORKDIR /app
ARG VERSION
ENV VITE_APP_VERSION=$VERSION
COPY ui/package*.json ./
RUN npm ci
COPY ui/ .
RUN npm run build

# Stage 2: serve
FROM nginx:alpine
COPY --from=builder /app/dist /usr/share/nginx/html
ARG VERSION
LABEL org.opencontainers.image.version="$VERSION"
LABEL org.opencontainers.image.title="ui"
LABEL org.opencontainers.image.source="https://github.com/cpolzer/wrkflw"
EXPOSE 80
```

### Dockerfile Changes (api-service and worker-service)

Add to the final stage of each existing Dockerfile:

```dockerfile
ARG VERSION
LABEL org.opencontainers.image.version="$VERSION"
LABEL org.opencontainers.image.title="<service-name>"
LABEL org.opencontainers.image.source="https://github.com/cpolzer/wrkflw"
```

Note: `ARG` must be re-declared in each stage that uses it (Docker build arg scoping).

### gradle.properties Change

```properties
# Before: (empty file)
# After:
version=0.1.0
```

Gradle project property `version` is automatically available to all subprojects. The `version = "0.1.0-SNAPSHOT"` lines in `apps/api-service/build.gradle.kts` and `apps/worker-service/build.gradle.kts` are removed; they inherit the root value.

### SARIF Upload (optional enhancement, included in tasks)

Trivy's SARIF output can be uploaded to GitHub's Code Scanning tab via `github/codeql-action/upload-sarif@v3`. This surfaces CVE findings as Security Alerts on the repository without requiring any external service.
